## Phase Scope Note

Phase 4 is a backend hardening phase. Its requirements — SEC-01, SEC-02, SEC-03, CONT-02, CONT-03, OCR-01, OCR-02 — are all server-side changes (auth bypass removal, CORS lockdown, production requirements file, .dockerignore, Tesseract integration). No new UI components or pages are introduced.

This design contract exists to:
1. Document the existing design system so the executor does not deviate from it
2. Specify the one UI surface that IS affected: auth error message copy (SEC-01 changes the 401 response body, which surfaces in LoginPage.tsx and RegisterPage.tsx error divs)
3. Confirm the design system is stable and requires no changes for this phase

## Color

Sourced from `frontend/src/styles/tokens.css`:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FDFBF7` (`--bg-base`) | Page background |
| Secondary (30%) | `#ffffff` (`--bg-surface`) + `#F5F0EB` (`--bg-raised`) | Cards, form panels, sidebar, elevated surfaces |
| Accent (10%) | `#C2410C` (`--accent-primary`) | Primary CTAs and active states only — see reserved list |
| Destructive | `#B91C1C` (`--cost-negative`) | Destructive actions, critical errors |

Accent (`--accent-primary`) reserved for:
- Primary button fill (`.primary` variant in Button.module.css)
- Active/selected navigation items
- Focus ring (`:focus-visible` outline)
- `::selection` highlight

Do not add new uses of `--accent-primary` as decoration in this phase.

Semantic colors (`--cost-positive`, `--cost-warning`, `--cost-negative`) are reserved for budget status comparison rows only. The auth error div uses `--cost-negative` for error text — this is correct and consistent with the existing `LoginPage.module.css` pattern.

Note: The live token file uses a warm light theme, not the dark theme described in CLAUDE.md. Do not change the theme in this phase — match the existing light token values exactly.

## Interaction Patterns

No new interaction patterns are introduced in this phase. The following existing patterns remain authoritative:

**Error states (inline, not modal):**
- Auth errors render inline in the form card as a styled `<div>` with `--cost-negative` color
- No modal interruptions — consistent with CLAUDE.md: "Calm, inline, in `--cost-negative`. No modal interruptions for recoverable errors."
- The error div disappears on next submit attempt (`setError('')` called in `handleSubmit`)

**Loading states:**
- Button text changes to loading copy ("Signing in...") and `disabled` attribute is set
- No spinner — button text change is the loading indicator, consistent with existing pattern

**Focus states:**
- All interactive elements use `:focus-visible` with `outline: 2px solid var(--accent-primary); outline-offset: 2px` (defined globally in `global.css`)
- Do not override focus styles in this phase

**Hover states:**
- Primary button hover: background transitions from `--accent-primary` to `--accent-warm` (#D4956A) via `--transition-fast` (150ms ease)
- Secondary button hover: background fills to `--bg-raised`, border to `--text-muted`

## Accessibility

Existing requirements remain in force (from CLAUDE.md):
- Minimum contrast 4.5:1 for all body text — already satisfied by existing token values
- `:focus-visible` ring already defined globally — no changes needed
- Icon buttons must have `aria-label` — not applicable to Phase 4 (no new icon buttons)
- Auth error messages must be associated with the form, not just visually present — ensure error div has `role="alert"` if not already present
