THREE changes across three files:

**1. ScheduleTimeline.tsx — Fix line 140 data pass**

Change line 140 from:
```tsx
<CookingGantt timeline={[...schedule.timeline, ...(schedule.prep_ahead_entries ?? [])]} totalDurationMinutes={schedule.total_duration_minutes} />
```
To:
```tsx
<CookingGantt timeline={schedule.timeline.filter((e) => !e.is_prep_ahead)} totalDurationMinutes={schedule.total_duration_minutes} />
```

This is the root cause fix. The current code concatenates prep-ahead entries into the timeline, but prep-ahead entries have offset values that break chart positioning. Only day-of steps should appear on the Gantt.

**2. CookingGantt.tsx — Add defensive filter + fix bar rendering**

a) In the `lanes` useMemo, add a defensive filter at the top to exclude any `is_prep_ahead` entries that slip through:
```tsx
const dayOfTimeline = timeline.filter((e) => !e.is_prep_ahead);
```
Then use `dayOfTimeline` instead of `timeline` in the rest of the `lanes` memo AND the `recipeColorMap` memo.

b) Also update the `hasPrepAhead` memo to use `dayOfTimeline` — but actually, since we are now filtering OUT prep-ahead entries before they reach the Gantt, `hasPrepAhead` will always be false. Remove the `hasPrepAhead` memo entirely, and remove the legend block at the bottom that depends on it (lines 175-181). The prep-ahead legend belongs in ScheduleTimeline, not in the Gantt chart for day-of steps.

c) The bar positioning math (lines 130-141) is correct — `leftPct = (time_offset_minutes / totalDurationMinutes) * 100` and `rawDurPct = (duration_minutes / totalDurationMinutes) * 100`. The issue was polluted data, not bad math. Keep the bar math as-is.

d) Add `tabIndex={0}` to each `.barGroup` div for keyboard accessibility (line 145 area):
```tsx
<div
  key={task.step_id}
  className={styles.barGroup}
  style={{ left: `${leftPct}%`, width: `${durPct + bufferPct}%` }}
  title={task.action}
  tabIndex={0}
>
```

e) Add conditional bar label visibility — hide label when bar is very narrow. Replace the barLabel span:
```tsx
{(durPct + bufferPct) > (24 / barAreaWidth * 100) && (
  <span className={styles.barLabel}>{oneWord(task.action)}</span>
)}
```
Actually, since we don't have barAreaWidth at render time in a clean way, use a CSS approach instead. Keep the barLabel span as-is and add a CSS rule to handle overflow (the existing `overflow: hidden; text-overflow: ellipsis` on `.bar` already handles this). The 24px minimum label width is better enforced via CSS `min-width` on the label container. No JS change needed here — the existing CSS already clips overflowing labels.
